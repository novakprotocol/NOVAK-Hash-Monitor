NOVAK Cryptography 101 ‚Äî Hashes, HVET, and Execution Receipts

This page is for engineers and auditors who want a clear, practical explanation of how NOVAK uses cryptographic hashing to guarantee execution integrity ‚Äî without needing a PhD in cryptography.

1. What is a cryptographic hash?

NOVAK relies on standard cryptographic hash functions (e.g., SHA-256). A hash function takes any input (1 byte or 1 GB) and returns a fixed-size output (for SHA-256, 256 bits). We treat the hash as a fingerprint of the data.

We rely on four key properties:

  • Deterministic ‚Äî same input, same hash every time.
  • One-way ‚Äî given the hash, it is infeasible to recover the input.
  • Collision-resistant ‚Äî infeasible to find two different inputs with the same hash.
  • Avalanche effect ‚Äî 1-bit change in input ‚Üí completely different hash.

In other words: if the hash changes, the underlying data has changed. NOVAK uses this to detect any tampering before automated decisions are allowed to execute.

2. The problem NOVAK fixes

TLS, disk encryption, logging, and even blockchain mostly confirm that something was sent, stored, or recorded correctly at time T. They do not guarantee that:

  • the inputs were correct and unmanipulated,
  • the governing rules were the intended version, or
  • the outputs weren‚Äôt silently altered before execution.

NOVAK adds the missing piece: proof-before-action. Systems must prove that the right data and the right rules produced the right output ‚Äî or execution is blocked.

3. HVET ‚Äî Hash-Verified Execution Trace

NOVAK defines a Hash-Verified Execution Trace (HVET) as a cryptographic commitment to the rule, the input data, the output, and the time of a decision.

Conceptually:

HR = SHA-256(rule_blob)        // governing rule set
HD = SHA-256(input_data_blob) // attested input
HO = SHA-256(output_blob)     // decision / result
t  = ISO-8601 timestamp       // when this decision was made

HVET = SHA-256(HR || HD || HO || t)

Here || means concatenation. If any of these change ‚Äî rule version, input, output, or timestamp ‚Äî the resulting HVET changes. That‚Äôs the foundation of NOVAK‚Äôs execution integrity.

4. EIR ‚Äî Execution Identity Receipt

An Execution Identity Receipt (EIR) is NOVAK‚Äôs canonical record of a vetted automated decision. It packages the hashes and metadata into a tamper-evident receipt that can be audited later.

An EIR typically includes fields like:

eir_id        // unique receipt identifier
HR           // hash of governing rule set
HD           // hash of attested input data
HO           // hash of output/decision
timestamp    // when the decision was produced
hvet         // HVET = H(HR || HD || HO || timestamp)
recorded_at  // when it was written into the audit log
version      // e.g., "NOVAK-EIR-v1"

Think of an EIR as a cryptographic invoice for what a system decided, when, and under which rules.

5. RGAC ‚Äî Recursive Global Audit Chain

Individual receipts are powerful, but regulators and auditors also need history. NOVAK links EIRs into a Recursive Global Audit Chain (RGAC) using simple hash chaining. 

// pseudo-code
link_0 = SHA-256("NOVAK-GENESIS")

for each EIR_i in chronological order:
    link_i = SHA-256(link_{i-1} || hvet_i)

Each new link commits to the entire history behind it. If any past EIR or link is altered, the chain no longer verifies. This gives auditors a blockchain-style tamper-evident log, without needing global consensus or mining.

6. Safety Gate, PL-X, and PS-X (high level)

Before an automated decision is allowed to execute, NOVAK‚Äôs Safety Gate verifies not just hashes, but also the sanity and legitimacy of the decision context.

  • PL-X ‚Äî Physical Layer Drift: detects corrupted or impossible values (sensor glitches, overflow, malformed encodings, out-of-range measurements, etc.).
  • PS-X ‚Äî Psycho-Social Manipulation: detects manipulative or policy-violating instructions, prompts, or rule changes that could cause harmful but superficially ‚Äúvalid‚Äù decisions.

If PL-X or PS-X flags a violation, execution is blocked and no EIR is issued. Instead, the system records an attempted action with a failed safety check.

7. How this differs from blockchain, TLS, and logging

  • TLS secures the channel between endpoints, but not the correctness of the rules or outputs on either end.
  • Logging records events after they happen ‚Äî often after damage is done.
  • Blockchain gives you a decentralized ledger of transactions, but does not guarantee that each transaction was computed according to the intended regulatory or safety rules.

NOVAK is specifically about execution integrity: proving that each automated decision obeyed the right rules, used the right data, produced the claimed output, and passed safety checks ‚Äî before it was allowed to affect the real world.
